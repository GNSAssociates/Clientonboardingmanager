### M5.5 — AI agents: Compliance Reviewer + Risk Assessor

**Compliance Reviewer** (`packages/ai/src/agents/compliance-reviewer.ts`):
- Model: Opus (`complex`) — depth of analysis required for regulatory work
- HITL: `always` (ComplianceOfficer) — **all compliance decisions require human sign-off** per MLR 2017
- Confidence threshold: 0.70
- 3 validators: red-flags require review, high-risk requires review, non-compliant has required actions
- Input: full CDD profile (KYC status, PEP/sanctions flags, documents, risk rating)
- Output: `overallCompliant`, `amlRisk`, `gaps`, `redFlags`, `recommendations`, `requiredActions`

**Risk Assessor** (`packages/ai/src/agents/risk-assessor.ts`):
- Model: Opus (`complex`) — multi-dimensional scoring
- HITL: `below_threshold` at 0.75 (ComplianceOfficer) + high-risk always requires review
- 3 validators: high-risk requires review, high-risk requires EDD, score-matches-rating
- Input: client type, SIC codes, jurisdiction, PEP/sanctions flags, source of funds
- Output: `riskRating` (low/medium/high), `overallScore` (0-100), factor breakdown, DD level, next review period


## Architecture notes

### MLR 2017 compliance
All CDD and compliance-gate decisions write to `audit_logs` (append-only). The `compliance_gates` table provides a tamper-evident record of what was approved, when, and by whom. REVOKE prevents Supabase auth roles from mutating records directly.

### KYC result flow (production)
In production, Amiqus sends a webhook to `/api/webhooks/kyc` (to be wired in M12 n8n) which calls `updateKycResult`. Staff can also manually poll via the PATCH endpoint.
