# M14 — Observability, Deployment Runbook & Load Validation

## 1. Observability Stack

### Error Tracking — Sentry
- SDK: `@sentry/nextjs` wired via `apps/web/instrumentation.ts`
- `SENTRY_DSN` environment variable controls activation (disabled in local dev if absent)
- PII scrubbing: `beforeSend` hook strips cookies; pino `redact` paths strip tokens/PII
- Trace sample rate: 10% in production, 100% in staging

### Structured Logging — pino
- Exported from `packages/core/src/observability.ts` as `logger` and `createChildLogger`
- Log level: `info` in production, `debug` in development (controlled by `LOG_LEVEL` env var)
- Audit events: `auditLog()` emits `audit: true` flag — can be routed to separate sink (e.g. Supabase table, CloudWatch Logs)
- pino-pretty for local dev; JSON for production

### OpenTelemetry (future)
- Instrumentation hook (`register()`) in `apps/web/instrumentation.ts` is the entry point
- Extend with `@opentelemetry/sdk-node` + OTLP exporter for distributed tracing when needed

## 2. Deployment Runbook

### Prerequisites
- Vercel account with `gns-onboarding` project created
- Supabase EU project with connection string in env vars
- GitHub Actions secrets: `VERCEL_TOKEN`, `SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY`, `SENTRY_DSN`

### First-time setup

```bash
# 1. Install dependencies
pnpm install

# 2. Copy env template
cp apps/web/.env.example apps/web/.env.local
# Fill in: SUPABASE_URL, SUPABASE_ANON_KEY, SUPABASE_SERVICE_ROLE_KEY,
#          ENTRA_CLIENT_ID, ENTRA_CLIENT_SECRET, ENTRA_TENANT_ID,
#          DROPBOX_SIGN_API_KEY, AMIQUS_API_KEY, COMPANIES_HOUSE_API_KEY,
#          XERO_CLIENT_ID, XERO_CLIENT_SECRET, QBO_CLIENT_ID, QBO_CLIENT_SECRET,
#          GRAPH_MAILER_TOKEN, GRAPH_MAILER_FROM, SENTRY_DSN, N8N_WEBHOOK_SECRET

# 3. Run migrations
cd packages/db && npx drizzle-kit migrate

# 4. Seed entities (GNS, LLP, GXY)
cd packages/db && npx tsx src/seed/entities.ts

# 5. Build + typecheck
pnpm -r typecheck && pnpm -r build
```

### Deployment (CI/CD)

GitHub Actions workflow (`.github/workflows/deploy.yml`):
1. `pnpm install --frozen-lockfile`
2. `pnpm -r typecheck`
3. `pnpm -r lint`
4. `pnpm -r test`
5. `npx drizzle-kit migrate` (runs against `DATABASE_URL` set in CI secrets)
6. `vercel deploy --prod` (via Vercel CLI)

### Database migrations
- All migrations are forward-only SQL files in `packages/db/migrations/`
- Run `npx drizzle-kit migrate` before deploying new app version
- Never edit existing migration files — create a new one
- `_journal.json` tracks applied migrations

### Rollback procedure
1. Revert app deployment to previous Vercel deployment (instant in Vercel UI)
2. Database: create a new compensating migration if schema was changed (never run reverse migration in prod)
3. Notify team via Slack `#ops` channel

## 4. Load Validation Notes

### Approach
Full load testing with k6/Playwright is out-of-scope for initial deployment (small-practice scale). The following sizing guidance applies:

### Expected scale
- Active cases: ~50–200 concurrent per entity (3 entities = ~600 max)
- Staff users: ~10–20 per entity
- Client portal users: ~200–500 total
- Peak: morning batch (09:00–11:00 UK) when staff action cases

### Vercel + Supabase sizing
- **Vercel**: serverless functions auto-scale; no warm-up needed for RSC
- **Supabase**: Free/Pro tier handles up to 500 concurrent connections; connection pooling via PgBouncer is auto-enabled
- **Drizzle**: connection pool size = `DATABASE_POOL_SIZE` env var (default 10 per function instance)

### Performance targets
| Endpoint | p95 target |
|----------|------------|
| Case list (`GET /api/v1/cases`) | < 200ms |
| Case detail + documents | < 500ms |
| AI agent trigger (async) | < 100ms (just creates a run record) |
| Completion report generation | < 2s |
| Ledger snapshot pull (Xero) | < 5s (external API bound) |

### Monitoring
- Vercel Analytics: built-in Core Web Vitals
- Supabase Dashboard: query performance, connection pool usage
- Sentry: error rate, transaction traces
- pino logs → Vercel Log Drains → Datadog/CloudWatch (optional)
