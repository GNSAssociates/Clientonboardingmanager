### M6.2 — Clearance letter template + Mailer port

**Clearance request template** (`packages/core/src/templates/clearance-request.ts`):
- ICAEW Code of Ethics R320.7-compliant letter
- Variable sections: entity branding, client details, previous firm, response deadline
- Asks the four standard clearance questions (outstanding fees, professional reasons, client integrity, records availability)
- Configurable response deadline (default 14 days)

**Mailer adapters** (`packages/integrations/src/adapters/`):
- `mailer-graph.stub.ts` — Microsoft Graph `sendMail` primary adapter; stub logs when `GRAPH_MAILER_TOKEN` absent
- `mailer-smtp.stub.ts` — SMTP fallback via nodemailer (runtime dynamic import, no compile-time dep); stub logs when `SMTP_HOST` absent
- Both registered in `adapters`: `mailer` = Graph (primary), `mailerFallback` = SMTP

### M6.3 — Clearance service + repository + API routes

**Repository** (`packages/db/src/repositories/clearance.ts`):
- `insertClearanceRequest`, `getClearanceRequestById`, `getClearanceRequestByCaseId`
- `listClearanceRequestsByCase`, `updateClearanceRequest`
- `insertClearanceFollowup`, `listFollowupsByRequest`, `listFollowupsByCase`

**Service** (`packages/core/src/services/clearance.ts`):

| Function | What it does |
|---|---|
| `sendClearanceRequest` | Render template → store generated_document → create clearance_request → send via mailer |
| `recordClearanceResponse` | Record outcome (clear/issues/refused/no_response) + update status to `received` |
| `sendFollowup` | Append followup row, increment chase number, update next chase date, resend via mailer |
| `getClearanceSummary` | Return all requests + nested followups for a case |

**API routes:**
| Route | Method | Description |
|---|---|---|
| `/api/v1/cases/:id/clearance` | GET | Full clearance summary for a case |
| `/api/v1/cases/:id/clearance` | POST | Send clearance request (renders template + emails) |
| `/api/v1/cases/:id/clearance/response` | POST | Record response from previous firm |
| `/api/v1/cases/:id/clearance/followup` | POST | Send a chase and log the followup |

### M6.4 — Prev-Accountant Communicator AI agent

`packages/ai/src/agents/prev-accountant-communicator.ts`:
- Model: Opus (`complex`) — drafts require professional and legally-sound language
- HITL: `first_then_auto` (Manager) — first contact always reviewed; routine chases auto-approved at ≥0.75 confidence
- Contact purposes: `initial_clearance`, `followup_chase`, `handover_request`, `records_request`, `general`
- Tone control: `formal` / `firm` / `urgent` (escalates language and deadline language)
- 3 validators: low-confidence auto-send rejected; valid response date required; body must be HTML
- Output: typed `subject`, `body` (HTML), `keyPoints[]`, `requestedResponseDate`, `suggestedFollowupDays`

## Mailer configuration

| Variable | Purpose |
|---|---|
| `GRAPH_MAILER_TOKEN` | OAuth2 access token for Microsoft Graph `sendMail` |
| `GRAPH_MAILER_FROM` | UPN/email to send from (required with Graph) |
| `SMTP_HOST` | SMTP server hostname (activates SMTP adapter) |
| `SMTP_PORT` | Default 587 |
| `SMTP_USER` / `SMTP_PASS` | SMTP credentials |
| `SMTP_FROM` | From address (defaults to `SMTP_USER`) |
| `SMTP_SECURE` | `"true"` for TLS (port 465) |

Without any of these, both adapters log to console (stub mode) — safe for local development.

## Clearance state flow

```
sendClearanceRequest()  →  status: sent  →  nextChaseAt: T+3
sendFollowup()          →  status: chased →  nextChaseAt: T+7 / T+14 (n8n wires chasers in M12)
recordClearanceResponse() → status: received, outcome: clear|issues_raised|refused|no_response
```

Automatic chasers at T+3/T+7/T+14 will be wired via n8n in M12.
