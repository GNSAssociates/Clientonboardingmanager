- `generated_documents` table — stores generated letters keyed by type + templateKey + storagePath + templateData JSONB
- `esign_envelopes` table — one envelope per generated doc; tracks provider, providerRef, signers JSONB, status, sentAt/signedAt/voidedAt
- `esign_events` table — append-only webhook log (no UPDATE/DELETE)
- New enums: `generated_doc_type`, `esign_status`

### M4.4 — Document generation service + API routes
`packages/core/src/services/generated-docs.ts`:
- `generateLetter(session, input)` — renders template → HTML, stores row in `generated_documents`, emits `document.generated` event
- `sendForSigning(session, input, esignProvider)` — looks up generated doc, calls `esignProvider.createEnvelope()`, stores `esign_envelopes` row, emits `esign.sent`
- `voidEnvelope(session, envelopeId, esignProvider)` — calls provider void + updates DB status
- `handleEsignWebhook(rawInput)` — processes Dropbox Sign callback, maps event_type → status, updates envelope + appends `esign_events` row
- `listGeneratedDocsForCase(session, caseId)` — returns `{ docs, envelopes }`

New repository: `packages/db/src/repositories/generated-docs.ts` — CRUD for `generated_documents`, `esign_envelopes`, `esign_events`

API routes:
| Route | Method | Handler |
|---|---|---|
| `/api/v1/cases/:id/generate` | POST | `generateLetter` |
| `/api/v1/cases/:id/esign` | GET | `listGeneratedDocsForCase` |
| `/api/v1/cases/:id/esign` | POST | `sendForSigning` |

## Architecture notes

### Template → PDF flow
```
Handlebars template + data
        ↓ renderTemplate()
    HTML string
        ↓ wrapHtml()      (A4 print-ready shell)
    stored as .html in dev
        ↓ Gotenberg/Chromium (in production)
    PDF binary → Supabase Storage
```
In production, swap `storagePath` from `.html` → `.pdf` and add a Supabase Edge Function or an internal API route that pipes the HTML through Gotenberg.

### Webhook idempotency
`esign_events` is append-only with no unique constraint — duplicate Dropbox Sign retries are safe (the event is logged, and envelope status updates are idempotent). In production, add a `(envelope_id, event_type)` unique index or store the provider's `event_hash` to deduplicate.

### Webhook verification (TODO production hardening)
The current webhook handler does not verify the `X-Hellosign-Signature` HMAC. Wire this in M13 (security hardening) by comparing `HMAC-SHA256(apiKey, payload)` against the header before processing.
