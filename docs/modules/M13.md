# M13 — Compliance, Audit & Retention Hardening + Security Review

## Overview

This module hardens the platform to meet ICAEW, HMRC, and MLR 2017 requirements for data retention, audit completeness, and information security.

---

## 1. Data Retention

### Policy
- Retention period: **6 years + current year** (ICAEW/HMRC default)
- Mechanism: soft-delete (`deleted_at` timestamp) at application layer, `retention_jobs` table for scheduled hard-purge
- `retention_jobs` table (migration `0016_retention_jobs.sql`):
  - Populated on every soft-delete with `scheduled_at = deleted_at + 7 years`
  - Executed by a weekly worker (n8n schedule or Supabase pg_cron)
  - Status: `pending → executing → completed | error`
- Documents stored in Supabase Storage: signed URLs expire after 24 hours; permanent deletion triggered by `retention_jobs` worker via Storage API

### Scope
All PII-bearing tables: `clients`, `client_contacts`, `documents`, `document_versions`, `generated_documents`, `kyc_checks`, `cdd_records`, `clearance_requests`, `communications`, `agent_runs`, `agent_approvals`, `review_tasks`, `review_findings`

## 2. Audit Log Hardening

### Properties
- Append-only: application never issues `UPDATE` or `DELETE` on `audit_logs`
- RLS: `authenticated` role has `INSERT` only; no `UPDATE`/`DELETE`
- Row structure: `id, entity_id, user_id, event_type, table_name, record_id, old_values, new_values, ip_address, user_agent, created_at`
- All state machine transitions, document uploads, AI decisions, compliance events, and HITL resolutions emit an audit event in the same transaction (transactional outbox)

### Coverage
| Domain | Events audited |
|--------|---------------|
| Cases | status transitions, create, archive |
| Documents | upload, verify, delete |
| Compliance | KYC result, risk rating, CDD completed |
| AI | agent run created, HITL resolved |
| Auth | login, logout, MFA events (via Supabase) |
| Admin | entity config changes, user role changes |

## 3. Security Controls

### Authentication
- Staff: Microsoft Entra ID SSO (OIDC) — MFA enforced by Entra policy
- Clients: Supabase Auth (magic link + password); MFA available
- Sessions: short-lived JWTs; server-side session via cookies (HttpOnly, Secure, SameSite=Lax)
- RLS enforced via `app.entity_ids` GUC set on every DB connection

### Authorisation
- RBAC + capability check in every service function (`authorize(session, capability)`)
- Admin routes gated by `session.isAdmin`
- API routes return 401/403, never leak entity/client data across tenants

### Input Validation
- All API inputs parsed with Zod before reaching service layer
- File uploads: MIME-type verification + max size enforced in Next.js API routes
- SQL injection: impossible — all queries via Drizzle ORM parameterised queries

### Secrets Management
- No secrets in repo (`.gitignore` covers `.env*`)
- Vercel environment variables for all API keys/tokens
- Integration OAuth tokens encrypted at rest (pgsodium / Supabase column encryption)
- `N8N_WEBHOOK_SECRET` used for HMAC-SHA256 signed n8n webhooks

### Encryption
- Data at rest: Supabase managed encryption (AES-256)
- Data in transit: TLS 1.2+ enforced by Vercel + Supabase
- Document storage: Supabase Storage (server-side encryption)
- Sensitive columns (OAuth tokens): encrypted with pgsodium before storage

### Content Security
- AI prompts: PII minimised — agent inputs use IDs, not raw names/addresses where possible
- Anthropic DPA signed; `X-No-Training` header on all Claude API calls
- No PII logged to Sentry (scrubbed by before-send hook)

### Rate Limiting
- Vercel edge rate limiting on `/api/v1/*` routes
- Supabase Auth rate limiting on auth endpoints
- n8n webhooks: IP allowlist + HMAC signature verification

## 4. OWASP Top 10 Checklist

| # | Vulnerability | Control |
|---|---------------|---------|
| A01 | Broken Access Control | RBAC + capability checks + RLS + `isAdmin` gates |
| A02 | Cryptographic Failures | TLS everywhere; pgsodium for OAuth tokens; Supabase AES-256 at rest |
| A03 | Injection | Drizzle ORM parameterised queries; Zod input validation |
| A04 | Insecure Design | Hexagonal ports; test coverage; capability-based auth |
| A05 | Security Misconfiguration | Environment variables only; no debug endpoints in prod |
| A06 | Vulnerable Components | `pnpm audit` in CI; Dependabot alerts |
| A07 | Auth Failures | Entra SSO + Supabase Auth; HttpOnly cookies; MFA |
| A08 | Software/Data Integrity | HMAC-signed webhooks; Zod validation on all inputs |
| A09 | Logging/Monitoring | Sentry + pino structured logs + OpenTelemetry traces |
| A10 | SSRF | No user-controlled URLs passed to server-side fetch |

## 5. MLR 2017 Compliance Notes

- CDD records retained for minimum 5 years from end of business relationship (§40)
- Risk assessments reviewed when triggered by client activity change
- Suspicious Activity Report (SAR) process: documented as staff SOP (not automated)
- AML supervisor registered on entity config (`amlSupervisor` field)
- All KYC/CDD decisions made by human (ComplianceOfficer) with AI advisory output only
